# 横断面戴帽道路模型设计

## 背景

横断面戴帽用于把道路中线、纵断面竖曲线设计高程和横断面路基模板组合起来，按不同桩号范围生成三维道路模型。当前 RoadProto 已具备道路中线实体、纵断面拉坡图、竖曲线实体和独立路基模板实体；本功能在这些能力之上新增道路模型实体，不把业务规则写入 UI 或命令回调。

## 目标

- 新增“横断面戴帽”命令，选择道路中线后配置不同桩号范围使用的路基模板。
- 表格行引用已有 `DnSubgradeTemplateEntity`，不在戴帽表格中内嵌模板参数。
- 表格行顺序表示优先级，越靠上的行优先级越高。
- 按道路中线平面、竖曲线高程和路基模板部件生成三维道路模型。
- 新增“编辑道路模型”命令和道路模型双击编辑入口，打开同一个 WPF 对话框修改模板范围并重新生成模型。

## 范围

第一版实现线框道路模型：每个路基模板部件至少生成对应三维线，不生成实体面、土方体、路面结构层实体或算量结果。未被戴帽表格命中的中线桩号段不生成模型。表格行可以短于或长于道路中线，生成时只使用与道路中线和竖曲线有效范围相交的部分。

## 架构

横断面戴帽归属 `CROSS_SECTION` 模块。

- `domain/cross_section` 新增道路模型数据结构和规则：
  - 戴帽表格行：起点桩号、终点桩号、模板 handle、模板名称。
  - 道路模型配置：道路中线 handle、竖曲线 handle、模型采样间距、戴帽表格行。
  - 三维部件线：模板引用、侧别、部件类型、部件序号、边界线序号和三维点序列。
  - 规则服务：模板优先级解析、采样桩号收集、路基模板展开和三维线生成。
- `application/cross_section` 新增道路模型创建/更新服务，组织 domain 规则，不直接依赖 ObjectARX。
- `cad_adapter/objectarx/cross_section` 新增：
  - `DnRoadModelEntity`：保存配置和生成结果，负责 DWG 持久化、三维线绘制、几何范围和变换。
  - 创建、编辑、handle 编辑和 WPF 回写命令入口。
  - WPF Bridge 请求/响应文件读写。
- WPF 新增横断面戴帽窗口，只负责展示和收集参数。

## 命令

- `RD_SECTION_ROAD_MODEL_CREATE`
  - 显示名称：横断面戴帽
  - 流程：选择道路中线，自动解析竖曲线，打开 WPF 窗口，点击“生成模型”后创建 `DnRoadModelEntity`。
- `RD_SECTION_ROAD_MODEL_EDIT`
  - 显示名称：编辑道路模型
  - 流程：选择道路模型实体，打开同一个 WPF 窗口，保存后更新实体。
- `RD_SECTION_ROAD_MODEL_EDIT_HANDLE`
  - 内部桥接命令，用于双击道路模型后按 handle 打开编辑窗口。
- `RD_SECTION_ROAD_MODEL_APPLY_DIALOG_FILE`
  - 内部桥接命令，用于读取 WPF 响应文件并创建或更新实体。

所有命令必须通过 `CommandRegistry` 注册，并在命令元数据中指向对应业务文档。

## 竖曲线解析

创建命令选择道路中线后，C++ 侧优先自动查找同一条道路中线的纵断面链路：

1. 查找 `DnProfileGradeGraphEntity`，其 `graphData().roadCenterlineHandle` 等于当前道路中线 handle。
2. 查找 `DnProfileVerticalCurveEntity`，其 `curveData().profileGraphHandle` 等于该拉坡图 handle。
3. 若找到唯一竖曲线，直接使用。
4. 若找不到或找到多个候选，提示用户选择竖曲线实体。

道路模型实体保存道路中线 handle 和竖曲线 handle。后续路线或竖曲线更新后的自动重建先不做，作为后续接入统一实体关系机制的扩展点。

## WPF 窗口

窗口名称为“横断面戴帽”。第一版只实现可用 tab `路基模板`，其他 tab 可以保留标题但禁用，不承载业务逻辑。

`路基模板` tab 包含：

- 模型采样间距，默认 `10.0`，必须为有限正数。
- 路基模板表格，列为：
  - 起点桩号
  - 终点桩号
  - 路基模板 handle
  - 路基模板名称
- 表格操作：
  - 新增行
  - 删除行
  - 上移
  - 下移
  - 选择模板
  - 生成模型
  - 取消

“选择模板”由 WPF 写出选择动作或请求关闭窗口，由 C++ ObjectARX Adapter 负责在 CAD 中选择 `DnSubgradeTemplateEntity` 并重新打开窗口；WPF 不直接读取 AutoCAD 实体。

## 模板优先级

戴帽表格不是普通分段表。对任一采样桩号，按表格行从上到下查找：

```text
if row.startStation <= station <= row.endStation:
    使用该行模板
    停止查找
```

靠上的行优先级更高。若没有任何行命中，该桩号不生成道路模型。行范围允许超出道路中线或竖曲线范围，超出部分忽略。起点大于终点、非有限桩号、空模板 handle 或无效模板实体属于无效输入。

## 采样规则

道路模型采样桩号由以下来源合并、排序、去重：

- 道路中线有效起终点。
- 竖曲线可计算起终点。
- 每个模板行与有效范围相交后的起终点。
- 按模型采样间距生成的中间桩号。
- 竖曲线关键桩号：起点、终点、PVI、BVC 和 EVC。
- 模板切换边界桩号。

最终只保留道路中线范围、竖曲线范围和模板命中范围共同覆盖的桩号。采样去重使用小容差，避免边界重复产生零长线。

## 三维线生成

每个采样桩号执行：

1. 从道路中线采样点插值得到中心平面点和切向方向。
2. 从竖曲线计算中心设计高程。
3. 以平面切向的法向作为横断面左右展开方向。
4. 读取命中的路基模板实体数据。
5. 对模板左右侧部件按从中线向外的几何顺序展开。
6. 部件实际宽度使用 `SubgradeTemplateRules::widthAtStation`。
7. 部件坡度使用 `SubgradeTemplateRules::slopeAtStation`；部件高度差继续作为模板已有参数参与断面点高程。
8. 生成该桩号处每个部件边界点的三维坐标。

相邻两个采样断面之间，只有当同一模板、同一侧、同一部件类型、同一部件序号和同一边界点序号连续命中时，才连接成三维部件线。模板切换处允许断线，不跨模板强行连接。第一版绘制三维线框，后续可以基于同一数据扩展为面、实体或导出普通 CAD 3D polyline。

## 实体持久化

`DnRoadModelEntity` 至少持久化：

- 版本号。
- 道路中线 handle。
- 竖曲线 handle。
- 采样间距。
- 戴帽表格行列表。
- 生成后的三维部件线列表。

实体绘制时按部件颜色绘制对应线。几何范围来自所有三维点。保存 DWG 后重开并 `REGEN`，道路模型和配置必须保持可编辑。

## 编辑流程

`RD_SECTION_ROAD_MODEL_EDIT` 选择 `DnRoadModelEntity` 后读取实体内配置，打开“横断面戴帽”窗口。用户修改表格、优先级或采样间距并点击“生成模型”后，C++ 使用当前道路中线、竖曲线和模板实体重新生成三维部件线并回写实体。

托管 Ribbon 插件监听双击事件。若双击对象 DXF 名称为 `DNROADMODELENTITY`，发送：

```text
RD_SECTION_ROAD_MODEL_EDIT_HANDLE <handle>
```

该命令进入同一编辑流程。

## 测试

核心测试覆盖不依赖 AutoCAD 的规则：

- 模板优先级解析：上方行覆盖下方行。
- 超出道路中线或竖曲线范围的模板行被裁剪。
- 未命中的桩号不生成。
- 采样桩号包含采样间距点、模板边界和竖曲线关键点。
- 道路中线样本、竖曲线高程和路基模板部件可以生成三维部件线。
- 模板切换处不跨模板连接。

源码约定测试覆盖：

- `CROSS_SECTION` 模块注册新增命令。
- C++ Ribbon model 包含横断面戴帽入口。
- 托管 Ribbon 插件包含按钮、`DNROADMODELENTITY` 双击转发和 WPF 命令类注册。
- Bridge 请求/响应字段包含道路中线 handle、竖曲线 handle、采样间距和表格行。

AutoCAD 图形界面手工验证：

- 创建命令选择道路中线后能自动或手动确定竖曲线。
- WPF 窗口可选择已有路基模板实体并调整表格顺序。
- 生成后图中出现三维道路模型部件线。
- 编辑命令和双击实体可重新打开同一窗口并保留上次配置。
- 保存 DWG 后重开并 `REGEN`，模型和配置保持正常。
